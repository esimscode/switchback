# Build a project case study

A case study converts a project into career proof. Structure:

1. **Problem** — the real situation that made the project worth doing. Concrete, no invented stakes.
2. **Solution** — what was actually built or done, in plain language a hiring manager can follow.
3. **Stack** — the technologies genuinely used. Take them from the project record; do not add tools that "should" be there.
4. **Architecture** — how the pieces fit together. Only describe what exists at the project's current status.
5. **Challenges** — real difficulties and how they were handled. These make the story credible.
6. **Outcome** — what the current, honest state is. Match the project's status label exactly: an In Progress project has an in-progress outcome ("working MVP with X and Y complete"), never a shipped-product claim.
7. **Next steps** — what's genuinely planned next.
8. **Career signal** — one or two sentences on what this project proves about the builder, tied to the target roles in the career profile.

Rules:
- Load the career profile first so the career signal aligns with the user's positioning.
- The project's status label caps every claim. Deployed ≠ Completed ≠ In Progress.
- Write in first person, calm and specific. No superlatives, no invented metrics or user counts.
- Mark the case study status as draft — the user reviews before publishing anywhere.

Persist with `create_project_case_study` when the user wants it kept.
