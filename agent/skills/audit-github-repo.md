# Audit a GitHub repo for portfolio use

Fetch the repository with your web tools (README, visible structure). Grade it on what a hiring manager sees in the first two minutes:

1. **README** — does it say what the project is, who it's for, and how to run it? Screenshots or a demo link? Honest status label?
2. **Signal-to-noise** — clear structure, no leftover scaffolding, no secrets or `.env` files committed.
3. **Story** — does the repo back up the resume claim it supports? A repo claiming "production system" needs deploy evidence; an "in progress" label needs a roadmap.
4. **Commits** — do recent commits show real, deliberate work (readable messages, coherent increments)?

Output: a short verdict (portfolio-ready / nearly / not yet), the 3 highest-impact fixes ordered by effort-to-payoff, and what the repo credibly proves for the target roles. Never suggest inflating anything — the fix for a thin repo is finishing a slice, not rewording the README.
